# Lambda の基本

## Lambda とは

AWS Lambda はサーバーを用意せずにコードを実行できるサーバーレスコンピューティングサービス。リクエストが来たときだけ起動し、使った分だけ課金される。

## コールドスタート

Lambda 関数が初めて呼び出されたとき、または一定時間アイドル状態だった後に呼び出されたとき、実行環境の初期化が発生する。これをコールドスタートと呼ぶ。

- **コールドスタートの流れ**: 実行環境の作成 → ランタイムの初期化 → ハンドラー外のコード実行 → ハンドラー実行
- **ウォームスタートの流れ**: ハンドラー実行のみ（実行環境が再利用される）

### コールドスタートの影響要因

- **ランタイム**: Python, Node.js は比較的高速。Java, .NET はランタイムの初期化に時間がかかる
- **メモリ設定**: メモリを増やすと CPU も比例して増加し、コールドスタートが短くなる
- **パッケージサイズ**: デプロイパッケージが大きいほどコールドスタートが長くなる
- **VPC 接続**: VPC に接続する場合、ENI（ネットワークインターフェース）の作成に追加時間がかかる

### コールドスタートの軽減方法

- **Provisioned Concurrency**: 事前にウォームな実行環境を確保しておく（追加コストあり）
- **メモリを増やす**: CPU も増加するため初期化が高速化される
- **パッケージサイズの削減**: 不要な依存ライブラリを削除する
- **Lambda SnapStart**: Java ランタイム向けの最適化。初期化済みのスナップショットからの起動

## タイムアウト

Lambda 関数のタイムアウトは最大 15 分（900 秒）。デフォルトは 3 秒。

- API Gateway 経由の場合、API Gateway 自体のタイムアウトが 29 秒のため、Lambda のタイムアウトを 29 秒以下に設定するのが一般的
- Bedrock など外部 API を呼び出す場合は、レスポンスに時間がかかるため 30 秒程度に設定する

## メモリ設定

- 最小 128 MB、最大 10,240 MB（10 GB）
- メモリに比例して CPU が割り当てられる（1,769 MB で 1 vCPU 相当）
- メモリを増やすと実行時間が短くなるため、コストが下がる場合もある

## 同時実行数

- デフォルトの同時実行数はアカウント・リージョンあたり 1,000
- **予約済み同時実行**: 特定の関数に同時実行数を予約できる
- **Provisioned Concurrency**: 事前にウォームな環境を確保（コールドスタート回避）

## Lambda のライフサイクル

1. **INIT フェーズ**: ランタイムの初期化、ハンドラー外のコード実行（DB 接続の確立等）
2. **INVOKE フェーズ**: ハンドラー関数の実行
3. **SHUTDOWN フェーズ**: 実行環境の破棄（一定時間アイドル後）

ハンドラー外で初期化したオブジェクト（DB 接続、SDK クライアント等）はウォームスタート時に再利用される。
