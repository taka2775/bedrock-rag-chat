# IAM ポリシーの評価ロジック

## ポリシーの種類

- **アイデンティティベースポリシー**: IAM ユーザー/ロール/グループにアタッチするポリシー
- **リソースベースポリシー**: S3 バケットポリシーや Lambda のリソースポリシーなど、リソース側に設定するポリシー
- **SCP（サービスコントロールポリシー）**: AWS Organizations で組織全体に適用する制限ポリシー
- **パーミッションバウンダリー**: IAM エンティティの権限の上限を設定するポリシー

## 評価フロー

IAM がリクエストを評価する順序:

1. **明示的な Deny の確認**: いずれかのポリシーに明示的な Deny があれば、即座にリクエストを拒否する。他のポリシーで Allow されていても Deny が優先される
2. **SCP の確認**: Organizations の SCP で許可されているか確認する。SCP で許可されていなければ拒否
3. **リソースベースポリシーの確認**: リソースベースポリシーで Allow されていれば、アイデンティティベースポリシーの Allow がなくてもアクセスが許可される（同一アカウント内の場合）
4. **パーミッションバウンダリーの確認**: バウンダリーが設定されている場合、バウンダリーとアイデンティティベースポリシーの両方で Allow されている必要がある
5. **アイデンティティベースポリシーの確認**: 明示的な Allow があればアクセスを許可する
6. **デフォルト拒否**: 上記のいずれでも Allow されなければ、暗黙的に拒否される

## Allow と Deny の関係

- **明示的な Deny は常に最優先**。どんなに Allow を設定しても、Deny があれば拒否される
- **明示的な Allow がない場合はデフォルト拒否**。IAM はデフォルトですべてのアクセスを拒否する
- **同一ポリシー内で Allow と Deny が競合した場合も Deny が勝つ**

## ポリシーの記述例

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:*",
      "Resource": "*"
    },
    {
      "Effect": "Deny",
      "Action": "s3:DeleteBucket",
      "Resource": "*"
    }
  ]
}
```

このポリシーでは S3 の全操作が許可されるが、`DeleteBucket` だけは明示的に拒否される。

## クロスアカウントアクセス

異なる AWS アカウント間のアクセスでは、**双方のポリシーで Allow が必要**:

- リクエスト元アカウント: アイデンティティベースポリシーで Allow
- リクエスト先アカウント: リソースベースポリシーで Allow

どちらか一方でも欠けているとアクセスは拒否される。
